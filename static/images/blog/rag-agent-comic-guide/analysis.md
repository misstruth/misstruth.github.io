# RAG 和 Agent 漫画图解内容分析

源文档：`content/blog/rag-agent-practical-guide.md`

风格：`doraemon`

章节数：5

## 核心主题

这篇文章要澄清 `RAG` 和 `Agent` 的边界：RAG 负责先把相关证据找回来，Agent 负责基于目标、上下文和工具把事情做下去。真实系统里二者常常不是二选一，而是先检索、再交接、再行动。

## 章节拆分

### 01 RAG 和 Agent 先分工

核心概念：
- RAG 解决模型不知道的问题
- Agent 解决知道以后怎么做的问题
- 中间关键是交接

视觉目标：建立整篇文章的角色和图例系统，读者一眼知道蓝色路线是 RAG，绿色路线是 Agent。

### 02 先检索，再行动

核心概念：
- RAG 把当前任务需要的上下文找出来
- Agent 基于上下文调用工具
- 上下文注入比把所有东西塞进大 prompt 更稳

视觉目标：用流程漫画说明从用户问题到证据召回，再到工具执行的链路。

### 03 三种上下文别混装

核心概念：
- 会话历史记录当前对话
- 记忆保存长期偏好
- RAG 拉取外部证据

视觉目标：用三个收纳盒/抽屉解释三种上下文容器的区别。

### 04 需求先看是在问，还是在做事

核心概念：
- 主要是回答问题时优先 RAG
- 主要是调用工具做事时优先 Agent
- 既要查资料又要执行时组合使用

视觉目标：用决策路口或选择板，把判断逻辑压成一张图。

### 05 RAG 底座和分块顺序

核心概念：
- 数据库选型不是 RAG 成败的唯一核心
- 切块、metadata、hybrid、rerank、上下文拼装更直接影响效果
- 先做干净语料和分块，再谈底座扩展

视觉目标：把数据库选择、分块策略、上线顺序画成生产检查清单。

## 生成约束

- 每张图为一页 16:9 横版 PNG。
- 每页 4 格，2x2 布局。
- 保持同一套角色外观：蓝白圆形导师角色、黄衣戴眼镜学生。
- 图片内中文要短，气泡单句不超过 12 字，标签不超过 6 字。
- 技术标签保留英文缩写：`RAG`、`Agent`、`prompt`、`rerank`、`metadata`。
- 每张图都要有标题面板、对话铺垫、图解面板、总结面板。
